# OpenClaw / 小龙虾工程建设手册

## 文档目的

这份文档不是源码讲解，也不是功能清单。

它的目标只有一个：把当前这个工程里真正值得抄走的设计能力、工程经验、方法论和组织方式，提炼成一套适合你在 `OpenClaw + 小龙虾` 场景里直接落地的建设方案。

你可以把它理解成：

- 给你自己的 agent 平台做产品设计和工程设计的参考底稿
- 给你的小龙虾喂的一份长期行为规范和运行策略
- 给后续研发工程拆模块、排优先级、控成本、提准确率的执行手册

这份文档会重点回答四个问题：

1. Claude Code 这类工程到底牛在哪里
2. 哪些东西最值得你抄，而且适合个人玩家
3. 怎么把这些能力迁移到你自己的 OpenClaw / 小龙虾体系里
4. 怎么让它变得更高效、更准确、更便宜、更稳定

---

## 一句话结论

这类工程真正牛的地方，不是“模型会调用工具”，而是它把 Agent 做成了一个完整的运行时系统。

不是一个 prompt 套壳，也不是一个聊天窗口加几个函数调用，而是一套带有：

- 统一输入入口
- 工具协议
- 权限系统
- 队列和并发控制
- 上下文压缩
- 成本统计
- 诊断系统
- 插件和技能扩展
- 远程执行通道
- 会话恢复和生产治理

的工程化操作系统。

你要养好自己的“小龙虾”，最应该学的不是某个具体 prompt，而是这套运行时设计。

---

## 一、这个工程里最值得抄走的能力

## 1. 把 Agent 做成“运行时”，不是“单轮问答”

这个工程的核心思路非常明确：

- 用户输入不是直接扔给模型
- 输入先经过统一入口处理
- 再进入消息队列、命令分流、上下文组装、权限检查、工具调度、结果回写
- 最后才形成一个完整的回合执行

这意味着产品不是“发一条消息，收一条消息”，而是“驱动一个会持续运行的工作循环”。

对你的小龙虾来说，这个认知非常关键：

- 它应该是一个持续执行的工作代理
- 它应该记得当前会话状态
- 它应该知道自己有没有任务在跑
- 它应该知道哪些输入要排队、哪些操作要中断、哪些行为需要确认
- 它应该具备本地执行、远程执行、恢复执行的能力

如果你只是做一个“接收 prompt -> 调模型 -> 回答”的壳，那它永远养不成生产级小龙虾。

---

## 2. 工具不是函数，而是“带治理信息的能力单元”

这个工程里的工具不是简单的 `callTool(name, args)`。

每个工具本质上都是一个完整能力对象，至少包含这些信息：

- 输入 schema
- 用户可理解的描述
- 是否只读
- 是否危险
- 是否允许并发
- 是否需要权限确认
- 是否需要额外校验
- 执行逻辑
- 结果格式

这套设计特别重要，因为它让系统可以在执行前就做出很多治理动作：

- 可以知道哪些工具适合并发
- 可以知道哪些工具必须串行
- 可以知道哪些工具要弹权限框
- 可以知道哪些工具适合给模型默认展示
- 可以知道哪些工具要被 deny rule 过滤掉

对 OpenClaw 的建议是：

- 不要把工具实现成一堆裸函数
- 必须定义统一 Tool Contract
- 工具能力、风险、展示策略、并发策略都写进 contract

建议最少包含以下字段：

```ts
type ToolSpec = {
  name: string
  description: string
  inputSchema: object
  isReadOnly: boolean
  isDestructive: boolean
  isConcurrencySafe: boolean
  shouldDefer?: boolean
  requiresPermission?: boolean
  validateInput?: (input, ctx) => Promise<ValidationResult>
  checkPermissions?: (input, ctx) => Promise<PermissionResult>
  execute: (input, ctx) => Promise<ToolResult>
}
```

这一步会直接决定你后面能不能做权限系统、工具排序、错误治理、并发调度。

---

## 3. 权限系统是生产能力，不是可选项

这个工程一个很成熟的地方，是它不是“工具都能跑”，而是“工具必须先过权限层”。

它做了三层事：

- 全局规则层：allow / ask / deny
- 工具级判断层：每个工具可自定义权限检查
- 交互层：需要确认时弹 UI，让用户永久允许、临时允许、拒绝

这里面的经验非常适合你：

### 你要抄的不是弹窗，而是权限治理模型

至少要有：

- `allow`
- `ask`
- `deny`

并且规则要支持：

- 按工具名
- 按工具参数模式
- 按域名
- 按目录
- 按远程服务

### 为什么这很重要

因为小龙虾最容易出事故的地方不是“回答错”，而是：

- 改错文件
- 在错误目录执行命令
- 提交危险 shell
- 意外把数据发到外网
- 读取不该读的内容

### 对 OpenClaw 的落地建议

最少要做这几个规则维度：

- `file_read`
- `file_edit`
- `file_write`
- `bash`
- `network_fetch`
- `remote_exec`
- `mcp:*`

规则存储建议支持三层：

- session 临时规则
- user 本地规则
- project 项目规则

这样你的小龙虾才能做到：

- 本轮先放行
- 以后这个项目默认放行
- 这个用户全局都放行

### 必须坚持的原则

- 不明确时默认 `ask`
- 危险操作默认 fail-closed
- 网络访问单独治理
- 文件写入和 shell 命令必须有独立权限面

## 4. 输入必须有统一入口

这个工程把直接输入、slash command、远程输入、排队输入都收敛到了统一入口。

这件事很重要，因为一旦输入入口分裂，你后面的行为就会越来越不一致：

- 有的走权限
- 有的不走权限
- 有的会进入历史
- 有的不会
- 有的能排队
- 有的不能

对你的小龙虾来说，必须建立一个统一的 `handlePromptSubmit` 层。

它至少负责：

- 空输入拦截
- 历史记录
- slash command 分流
- 图片/附件展开
- prompt mode / bash mode 区分
- 加入队列或立即执行
- 创建 abort controller
- 进入主查询流程

建议你直接把输入分成三类：

1. 用户工作输入  
   例如“修复这个 bug”“帮我看下日志”

2. 系统级命令  
   例如 `/doctor` `/config` `/model`

3. 远程/桥接输入  
   例如移动端、网页端、remote daemon 过来的消息

统一入口的好处是：

- 所有治理逻辑都有一个落点
- 后续加新交互方式不会把系统搞乱
- 更容易做审计、埋点、恢复和回放

---

## 5. 并发必须是受控并发，不是“想到就并发”

这个工程里一个很高级但很实用的经验是：

- 不是所有工具都能并发
- 只读、无副作用、并发安全的工具可以一起跑
- 有写入和上下文副作用的工具要串行

这非常值得抄。

### 适合并发的

- 搜索
- 读取文件
- 查看目录
- 获取只读状态
- 拉取诊断信息

### 不适合并发的

- 编辑文件
- 写文件
- 执行有副作用的 shell
- 提交代码
- 发布
- 删除、覆盖、移动

### 你应该怎么做

每个工具声明：

- `isReadOnly`
- `isConcurrencySafe`

调度器根据这个做分批执行：

- 并发安全批次一起跑
- 非并发安全的逐个跑

这会同时带来三种收益：

- 速度更快
- 冲突更少
- 结果更可解释

对个人玩家来说，这个性价比非常高，建议尽早做。

---

## 6. Query Guard 和队列系统非常值得抄

这个工程没有把“是否正在运行”做成一个脆弱的布尔值，而是做成了一个明确的状态机。

核心状态是：

- idle
- dispatching
- running

这看起来很细，但非常有价值。

### 为什么不该只用一个 `isLoading`

因为在真实工程里，最容易出 bug 的不是执行中，而是“异步链路刚刚开始但还没完全进入执行”的空窗期。

比如：

- 队列刚取出任务
- UI 还没更新
- 执行函数还没真正开始
- 用户又提交了一次

如果你只有一个简单布尔值，就会出现：

- 双提交
- 重入
- 队列错乱
- 重复执行

### 对你的建议

直接抄这个思想：

- `idle` 没任务
- `dispatching` 已出队，正准备执行
- `running` 真在执行

并把它做成一个同步状态机，而不是散落在 React state 里。

这会明显提升小龙虾的稳定性。

## 7. 上下文管理是成本控制核心，不是附属功能

这个工程对上下文管理非常认真，不只是拼接 prompt，而是把上下文当成生产资源来治理。

它做了这些很关键的事：

- 区分 system prompt、user context、system context
- 有 token warning threshold
- 有 auto compact
- 有 reactive compact
- 有 manual compact
- 有 compact 失败熔断

这是你必须重点吸收的。

### 对个人玩家最重要的现实

你的小龙虾如果不管上下文，后面一定会：

- 越跑越贵
- 越跑越慢
- 越跑越不准
- 到后期开始胡说八道

### 你应该建立的上下文结构

建议分成这几层：

1. `system prompt`
   定义总体身份、行为规范、输出风格、底层原则

2. `runtime context`
   当前模型、工具池、权限模式、当前工作目录、远程状态

3. `task context`
   当前任务目标、限制、当前计划、最近操作

4. `repo context`
   仓库结构、关键文件、约束、测试方式

5. `memory context`
   长期记忆、用户偏好、项目约定

### Compact 的正确理解

compact 不是简单摘要，而是：

- 用更低 token 成本保留当前任务的必要状态
- 丢弃已经完成的细枝末节
- 防止上下文窗口被历史垃圾吃光

### 对 OpenClaw 的建议

先做一个足够实用的版本：

- 统计当前上下文 token 使用量
- 达到阈值时自动触发 compact
- compact 后保留：
  - 当前任务目标
  - 已做动作摘要
  - 当前文件状态
  - 剩余风险和待办

再加两条防线：

- 连续 compact 失败时停止自动重试
- 用户手工触发 `/compact`

## 8. 成本统计要变成一等公民

这个工程很成熟的一点是，它不是只算“总花费”，而是把成本拆得很细：

- 总成本
- 每模型成本
- 输入 token
- 输出 token
- cache read
- cache write
- API duration
- wall duration
- 行改动量
- web search 请求数

这套方法非常适合你，因为你要的明确是：

- 更便宜
- 更高效
- 更准确

而这三个目标只有在指标化之后才能真正治理。

### 你应该立即统计的指标

- 每轮 token 消耗
- 每轮 wall time
- 工具执行耗时
- 模型耗时
- 每任务总成本
- 每模型成本分布
- cache 命中收益
- 代码产出效率

### 你应该如何用这些指标

#### 控便宜

- 简单任务走便宜模型
- 复杂规划走强模型
- 搜索和读取尽量并发
- 高频只读上下文做缓存
- 大历史及时 compact

#### 控准确

- 看失败任务对应的上下文长度
- 看错误是否发生在低成本模型
- 看权限拒绝是否导致任务被迫走歪

#### 控效率

- 看工具耗时占比
- 看是否有排队阻塞
- 看哪些任务被过度串行化

没有指标，小龙虾只能靠感觉养；有了指标，你才能真正调教。

## 9. 诊断系统非常值得做

`Doctor` 这类能力在个人项目里经常被忽略，但其实是高杠杆能力。

它的价值不在“展示信息”，而在于让你迅速区分问题到底出在哪一层：

- 配置错了
- 权限没开
- MCP 坏了
- 插件加载失败
- 环境变量脏了
- 目录状态有问题
- 锁文件冲突
- 上下文过大

### 你应该给 OpenClaw 做一个最小 Doctor

至少检测这些项：

- 当前模型配置
- API key / OAuth 状态
- 当前工作目录是否合法
- tools 是否注册完整
- MCP 服务连接状态
- 插件加载状态
- skills 扫描状态
- 权限模式
- token 使用告警
- 远程连接状态
- 输出长度限制

### 为什么它重要

因为 agent 系统最痛苦的不是“没功能”，而是“不知道为什么失败”。

Doctor 是把不可见系统状态变成可诊断状态。

## 10. 插件、技能、MCP 要分层，不要混成一锅

这个工程对扩展能力的组织方式很值得学习：

- 技能：偏 prompt/workflow
- 插件：偏工程扩展能力和附加命令
- MCP：偏外部工具和资源接入

你自己的 OpenClaw 也应该这么分。

### 技能适合放什么

- 开发流程模板
- 常见代码修复套路
- review 策略
- 文档生成套路
- 项目约束和最佳实践

技能本质是“行为模板”和“工作流提示器”。

### 插件适合放什么

- 自定义命令
- 钩子
- 私有工作流
- 集成行为
- 项目级增强

插件本质是“宿主扩展包”。

### MCP 适合放什么

- 外部数据源
- 检索系统
- 浏览器控制
- issue/PR 系统
- 自建平台服务

MCP 本质是“外部能力接入协议”。

### 你要避免的坑

- 不要把所有东西都做成 prompt
- 不要把所有东西都做成工具
- 不要让插件和 MCP 各自维护一套命令体系

最佳实践是：

- Prompt 型能力进技能
- 程序型扩展进插件
- 外部系统接入走 MCP

## 11. 技能热更新是个人玩家的大杀器

这个工程很聪明的一点是，技能文件变化后会自动刷新命令缓存。

这背后的价值是：

- 你可以把技能当成活文档
- 调 prompt、调 workflow 的成本非常低
- 不必每改一次都重启整个系统

对你的小龙虾来说，这是非常值得照抄的。

### 建议的技能文件结构

每个 skill 用一个 markdown 文件，带 frontmatter：

```md
---
name: fix-backend-bug
description: 修复后端 bug 的执行规范
allowed-tools:
  - file_read
  - file_edit
  - bash
when_to_use: 当用户要求排查和修复后端逻辑问题时使用
model: inherit
effort: medium
---

这里写技能正文...
```

### 你可以马上做的 skill 类型

- `bugfix`
- `review`
- `refactor`
- `write-tests`
- `summarize-repo`
- `release-check`
- `doctor-project`
- `openclaw-self-check`

个人玩家做 skill 的收益极高，因为这是“低代码强化 agent 行为”的最快路径。

## 12. 远程执行能力非常适合个人玩家进阶

你现在打开的 `direct connect` 相关实现，其实特别适合你后续做进阶版小龙虾。

它的思路是：

- 本地负责 UI 和交互
- 远程服务负责实际执行
- 通过 session 握手建立会话
- 用 websocket 收发消息
- 权限请求通过控制消息回到本地确认

这个模式的价值非常大：

- 本地体验不变
- 任务可以放远端跑
- 长任务和重任务更稳
- 可以多端接入
- 更容易做 daemon 化

### 适合你的使用场景

- 本地 Mac 做交互
- 远端 Linux 跑工程任务
- 编译、测试、爬取、索引等重活放服务器
- 本地保留权限确认和工作流控制

### 你应该怎么设计

拆成两个角色：

1. `Session Creator`
   负责建会话，拿 `session_id` 和 `ws_url`

2. `Session Manager`
   负责 websocket 生命周期、消息收发、权限请求、中断

不要把“建会话”和“跑通道”糊在一个类里，这个工程的拆法是对的。

## 13. 安全不是附属逻辑，而是基础设施

这个工程里有一个很实用的经验：敏感信息扫描在本地做，尽量别让秘密离开本机。

这件事你一定要学。

### 对 OpenClaw 的建议

在这些时机做本地 secret scan：

- 上传记忆前
- 发送远程内容前
- 提交外网前
- 建立长期知识库前

先做高置信度规则就够了，不要一上来搞太复杂：

- API keys
- token
- cloud credential
- private key
- PAT

### 再加两条底线

- 外网访问默认受控
- 本地文件越界访问默认受控

小龙虾一旦开始自动化，就必须把安全当成系统功能，而不是“提醒一下用户”。

## 二、面向 OpenClaw / 小龙虾的建设蓝图

## 1. 你应该建设的七层架构

建议你把系统拆成下面七层：

### 第一层：交互层

负责：

- REPL
- slash commands
- 多输入模式
- 输出渲染
- 通知
- 权限弹窗

目标：

- 让用户感觉自己在操作一个“工作台”
- 不是聊天框

### 第二层：提交入口层

负责：

- 输入预处理
- 历史记录
- 附件展开
- 命令分流
- 队列
- 创建本轮执行上下文

目标：

- 所有输入统一走这一层

### 第三层：Query Runtime 层

负责：

- system prompt 组装
- user context / system context 组装
- 发起模型请求
- 处理流式输出
- tool use loop
- auto compact

目标：

- 把“一轮 agent 执行”变成标准流水线

### 第四层：Tool Runtime 层

负责：

- 工具注册
- schema 校验
- 工具路由
- 并发与串行调度
- 工具结果规范化

目标：

- 让工具成为可治理能力单元

### 第五层：Permission & Safety 层

负责：

- allow / ask / deny
- 工具级权限
- 域名规则
- 路径规则
- secret scan
- sandbox / network policy

目标：

- 安全默认收紧，必要时按需放行

### 第六层：Extension 层

负责：

- skills
- plugins
- MCP
- hooks

目标：

负责：

目标：

- 让系统可控、可诊断、可优化

## 三、为了更高效：你应该优先抄的设计

## 1. 只读工具并发，写工具串行

这是性价比极高的性能优化。

### 直接收益

- 搜索更快
- 上下文收集更快
- 总体 turn latency 更低

### 风险也可控

- 只读工具冲突少
- 写工具不并发，避免状态乱掉

这是你最值得尽早做的性能优化之一。

---

## 2. 让长输出折叠显示

这个工程对读取类、搜索类、列表类命令做了折叠策略，这很实用。

对你的系统也建议这么做：

- 搜索结果折叠
- 大文件读取折叠
- 列目录折叠
- 长日志默认摘要 + 可展开

收益：

- UI 更清爽
- 上下文污染更少
- 用户感知更快

---

## 3. 输入排队而不是直接拒绝

当系统忙的时候，不要单纯说“请稍后重试”。

更好的方式是：

- 如果当前任务不可中断，就排队
- 如果当前任务可中断，就让用户选择中断或排队

这样用户体验会好很多，也更符合“工作台”的感觉。

---

## 4. 启动阶段做并行预取

这个工程会在启动阶段并行做一些准备工作，而不是串行慢慢等。

对 OpenClaw 你也可以照抄：

- 预热配置
- 预热身份态
- 预热插件元数据
- 预热 MCP registry
- 预热技能索引

目标不是“极限优化”，而是把冷启动延迟打下来。

---

## 四、为了更准确：你应该优先抄的设计

## 1. 工具调用前必须校验输入

模型不是严格程序，它会传错参数。

所以必须：

- schema 校验
- tool-specific validate
- 超界参数拦截
- 文件路径合法性检查
- 危险命令语义检查

不要相信模型“应该知道怎么传”。

---

## 2. 把 system prompt 和运行时上下文分开

这点非常关键。

system prompt 负责原则，运行时 context 负责事实。

不要把：

- 用户偏好
- 当前目录
- 当前模型
- 当前权限模式
- 当前任务计划

全塞进 system prompt。

正确做法是：

- system prompt 保持稳定
- runtime context 动态拼装

这样：

- cache 更容易命中
- prompt 更稳定
- 错误更容易定位

---

## 3. 必须有失败可解释性

每次拒绝或失败，尽量给出明确原因：

- 被权限规则拒绝
- 被工具校验拒绝
- 上下文过长
- 远程连接失败
- 插件加载失败
- MCP 不可用

不要只回一个 “failed”。

工程上这是非常值钱的，因为它会减少你自己后面排障的时间。

---

## 4. 给 agent 一个明确的行为边界

你的小龙虾要想准确，必须有清晰边界：

- 不确定时先搜索和读取
- 没验证前不下结论
- 没权限前不执行危险动作
- 不能为了完成任务而乱改文件
- 不能静默跳过失败

这部分不只是 prompt 问题，更是 runtime policy 问题。

---

## 五、为了更便宜：你应该优先抄的设计

## 1. 建立模型分层

不要所有事都上最贵模型。

建议分三层：

- 轻模型：搜索、分类、格式化、小修小补
- 中模型：常规开发和重构
- 强模型：规划、复杂修复、架构判断

如果 OpenClaw 后面支持模型路由，这是必须建设的。

---

## 2. 优先治理上下文，而不是盲目降模型

很多人为了便宜，第一反应是换便宜模型。

但实际更高收益的通常是：

- 减少无用历史
- 及时 compact
- 工具结果摘要化
- 大结果写磁盘只喂 preview
- 只把必要信息放进 prompt

这是这个工程里很强的一条隐含经验。

---

## 3. 对大结果做“持久化 + 摘要返回”

不要把所有工具原始输出都塞回模型。

正确做法：

- 大结果存盘
- 返回摘要、预览、路径
- 真需要时再读

收益很大：

- token 立刻下降
- 会话更稳定
- 模型更聚焦

这对日志、搜索结果、diff、长文档尤其重要。

---

## 4. 记录 cache read / cache write 收益

如果你的底层模型平台支持 prompt cache，一定单独统计：

- cache read token
- cache write token

因为这会直接影响你的成本模型和上下文策略。

## 七、给小龙虾直接喂的行为规范

下面这段你可以直接作为“小龙虾工程行为规范”的基础版本，后面按你自己的 OpenClaw 能力裁剪。

```md
# 小龙虾执行规范

你是一个面向工程生产的开发代理，不是普通聊天助手。

你的目标不是尽快回答，而是在成本可控、风险可控、结果可验证的前提下完成任务。

## 工作原则

1. 先理解再执行
在开始改动前，先读取必要上下文，确认目标、边界、相关文件和当前限制。

2. 优先低成本收集信息
优先使用搜索、读取、目录检查、状态检查等低成本只读动作，不要一上来做重推理和大范围改动。

3. 不确定时先验证
如果一个事实可能变化、可能不准确、或者会影响结论，先查证再输出。

4. 危险动作必须受控
涉及文件写入、批量修改、命令执行、网络访问、远程调用时，必须遵循权限规则；未明确允许时不要擅自执行高风险动作。

5. 尽量做局部、可回滚的修改
优先最小改动；避免无必要的大范围重写；每次修改都要能解释为什么做、影响了什么。

6. 输出要服务执行
优先给出可执行结论、明确建议、下一步动作，而不是泛泛而谈。

## 成本原则

1. 能用只读工具解决的问题，不要先走重模型推理
2. 能摘要返回的结果，不要整段灌回上下文
3. 会话过长时主动建议 compact
4. 对重复上下文尽量复用缓存或摘要

## 准确性原则

1. 不要把猜测说成事实
2. 没读过的文件不要假装读过
3. 没跑过的验证不要说已经验证
4. 如果存在多种可能，说明当前判断依据和风险

## 工程执行原则

1. 搜索、读取类操作可优先并发
2. 写入、编辑、提交、发布类操作默认串行
3. 大输出要先摘要再决定是否继续展开
4. 遇到权限拒绝、远程失败、上下文过长、工具不可用时，要明确说明阻塞点

## 对用户的协作方式

1. 直接推进，不要让用户替你做本可自动完成的事
2. 当存在高风险分叉时，用简洁方式说明选项和代价
3. 结果优先，废话最少，但关键原因必须说清
```

## 十、你现在最应该抄的，不是某个功能，而是这五个观念
